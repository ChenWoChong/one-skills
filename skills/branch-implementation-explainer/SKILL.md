# Branch Implementation Explainer

## 目标

这个 skill 不做 code review，不负责找 bug、回归风险、缺失测试或设计问题。它只做一件事：为一个分支生成或更新一份实现讲解文档，把这个功能到底怎么实现的讲清楚。

主交付物不是聊天里的解释，而是落到磁盘上的 Markdown 文档。

输出必须让用户在几分钟内回答下面这些问题：

- 这个分支具体要实现什么功能。
- 为了实现它，哪些核心数据结构被修改了。
- 请求、事件或任务从哪里进，经过哪些关键函数，最后落到哪里。
- 哪些函数是新加的，为什么必须加，分别承担什么职责。
- 哪些代码是真正核心代码，值得先看；哪些只是辅助代码。
- 这份讲解文档保存在哪里，后续应该基于哪份文档继续补充。

文档生成以后，用户应该能直接定位核心文件、顺着主链路读代码，并把这个实现讲给别人听。

## 什么时候用

以下场景应触发这个 skill：

- “帮我讲清楚这个分支具体怎么实现的。”
- “我运行前想先快速理解这个功能的核心代码。”
- “告诉我这个分支改了哪些结构、流程和函数，方便我继续改。”
- “不要给我 code review，只讲实现细节。”
- “给我一个能快速看懂这段功能实现的讲解。”
- “帮我生成一份这个分支的实现文档。”
- “基于之前那份实现文档继续更新。”

如果用户明确要 review、挑 bug 或判断测试是否充分，不要只用这个 skill，应改用专门的 review 技能。

## 工作流程

1. 先确认比较范围。
2. 确定文档路径，检查 `/Users/chenmoyin/Documents/impl-doc/` 下是否已有对应文档。
3. 看 diff 和关键文件，判断这个分支的功能目标。
4. 先找数据结构变化，再找主流程，再找新增函数。
5. 从真实入口往下串主链路，不按文件顺序复述 diff。
6. 把结果写成 Markdown 文档；如果已有文档，就在原文档上更新，而不是重新造一份平行版本。

## 规则

### 先确认比较范围

先搞清楚这个分支相对谁在变化。默认做法：

- 找 merge-base。
- 看 `git diff --stat` 和关键文件列表。
- 用提交信息、文档和测试名辅助判断功能目标。

如果比较基线错了，后面的结构讲解和流程讲解都会失真。

### 文档必须落到固定目录

这个 skill 的产物必须写到：

`/Users/chenmoyin/Documents/impl-doc/`

不要把文档写到仓库里的 `docs/`、临时目录、桌面或聊天回复里代替交付。

默认规则：

- 如果目录不存在，先创建目录。
- 默认文档名优先使用分支名，建议格式：`<branch-name>-implementation.md`。
- 分支名里如果有 `/`，替换成 `-`。
- 如果用户明确指定文档名，按用户要求；否则不要随意起一个难以复用的新名字。

### 修改和更新都基于已生成文档

如果 `/Users/chenmoyin/Documents/impl-doc/` 里已经存在这个分支对应的实现文档，后续补充、修改、扩写都应该直接更新这份文档。

不要每次都新生成一份 `v2`、`final`、`latest` 之类的平行文档，除非用户明确要求保留多版本。

更新已有文档时：

- 先读旧文档，保留已经正确的结构和内容。
- 在原有章节下补充新的结构图、代码片段、函数说明和流程说明。
- 如果原文档结构已经不清楚，可以重组章节，但仍然更新同一份文档，而不是另起一份。
- 让文档本身持续演进，成为后续讨论和修改的基准。

### 只讲实现，不做评判

这个 skill 的职责不是评审代码好坏，而是解释它怎么工作。

- 不输出 findings。
- 不主动评价“这里有 bug”“这里设计不好”。
- 不展开测试是否充分。
- 不把重点放在“还能怎么改”。

只有当某个实现选择本身会影响理解时，才说明它的取舍，例如“为什么把状态从 Session 挪到 LiveSession”。

最终聊天回复只需要简短说明：

- 文档生成或更新到了哪个路径。
- 这次主要新增或更新了哪些章节。

### 先讲结构，再讲流程，再讲函数

不要按文件顺序复述 diff。固定顺序如下：

1. 这个分支实现了什么功能。
2. 修改了哪些核心数据结构或状态关系。
3. 主链路怎么走。
4. 新增或修改的关键函数分别做什么。
5. 哪些代码最值得先看。

先讲“骨架”，再讲“血流”，最后讲“器官”。不要一上来就贴零散函数。

### 文档必须循序渐进

文档不是 diff 摘要，要让读者沿着一条自然路线读下去。

每份文档开头必须先给读者一个低成本入口：

- 先告诉读者这篇文档怎么读。
- 再用一句话说清楚这个分支做了什么。
- 然后给出改动范围和核心概念，避免读者一上来就掉进代码细节。

每个重要章节内部也要按这个顺序写：

1. 先用人话说结论。
2. 再给结构图、流程图或代码。
3. 最后解释为什么这么改、这段代码在链路里起什么作用。

不要把背景、结构、流程、代码、函数说明写成彼此孤立的段落。每一节都要自然承接上一节，让读者能从“为什么改”一路读到“具体怎么实现”。

### 必须给改动文件地图

文档早期必须有一节讲清楚这次分支主要改了哪些文件。

不要只列文件名，要按职责分组：

- 数据结构和状态模型相关文件。
- 主流程入口和 handler/service 相关文件。
- 持久化、索引、生命周期或资源管理相关文件。
- 测试、文档或配置相关文件。

每个文件都要用一句话解释它在这次功能实现里的角色。目标是让读者先知道代码地图，再进入具体代码。

### 必须给核心概念速查

如果这个分支引入或强化了新的领域概念、结构名、状态名、索引名或生命周期名，文档必须提前给一个“核心概念速查”。

这一节要讲人话，不要写成术语表堆砌：

- 这个概念是什么。
- 它原来由谁承担，或者为什么现在需要单独出现。
- 它和其他核心概念是什么关系。

后面再讲结构图、流程图和函数时，就可以复用这些概念，读者不会反复卡住。

### 数据结构变化必须讲修改前和修改后

只要分支为了实现功能修改了 struct、字段语义、索引关系、所有权、生命周期归属，就必须明确讲：

- 修改前结构是什么。
- 修改后结构是什么。
- 真正变化的是哪几个字段、索引或引用关系。
- 为什么这样改后，主流程更容易支撑这个功能。

如果只是说“新增了一个 struct”，但没有解释它替代了旧模型里的什么职责，这种讲解是不合格的。

这里不能只讨论，必须 show code：

- `修改前` 必须贴出具体代码，至少包含旧的核心 struct、关键字段、关键索引或关键入口函数。
- `修改后` 也必须贴出具体代码，至少包含新的核心 struct、关键字段、关键索引或关键入口函数。
- 如果变化点分散在多个文件，就分块贴代码，不要只贴一小段最表面的字段定义。
- 优先做“前后对照”讲解。让用户能直接看到代码从什么样子变成了什么样子。

代码越多越好，但前提是都服务于理解核心实现。宁可多贴 3 段真正关键的代码，也不要只写一句“结构从 A 变成了 B”。

### 必须给结构图

讲数据结构时，必须给结构图。优先用 Mermaid。

第一张图必须做到一眼看懂核心：

- 只保留理解这个功能绝对必要的对象和关系。
- 先让用户看懂“核心对象是谁、关系主干是什么、状态分层在哪里”。
- 不要把所有字段、所有索引、所有边角对象第一次就塞进一张图。

如果你自己判断“一张图虽然完整，但第一眼并不好懂”，就必须继续补图，不要硬撑一张大图。

图里至少表现这些信息：

- 核心 struct 或对象有哪些。
- 它们之间谁持有谁、谁引用谁、谁负责索引谁。
- 哪些是持久化状态，哪些是运行时状态。

如果功能核心是“一个状态被拆成两个层次”或“多个索引围绕一个对象重建”，图里必须画出来。

复杂结构默认用多张图递进降低理解难度，推荐这样拆：

1. 总览图：只画核心对象和主关系，一眼看懂。
2. 展开图：补充索引、绑定关系、运行时与持久化分层。
3. 演进图：如果修改前后差异很大，再单独画“修改前”和“修改后”两张图，或画一张“结构演进图”。

不要为了“只给一张图”牺牲可读性。多画两张简单图，通常比一张复杂图更容易理解。

每张图都要有明确目的，不能只是把同一堆关系重画一遍：

- 这张图是帮助用户先抓核心，还是补细节。
- 这张图是讲修改前，还是讲修改后。
- 这张图是讲静态结构，还是讲运行时绑定关系。

### 必须给主流程图

讲主链路时，必须给一个流程图。优先用 Mermaid。

流程图不是装饰，至少要覆盖一条真实路径：

`入口 -> 参数组装 -> 核心对象定位/创建 -> 关键状态变化 -> 外部副作用 -> 返回/持久化`

如果这个功能至少有两条同等重要的路径，例如 `new/load` 和 `prompt`，可以画两条，但不要把所有边角情况都塞进一张图。

### 必须给核心代码片段

不能只讲抽象概念，必须贴充分的关键代码片段，并解释为什么它们关键。

代码片段选择标准：

- 能代表新的核心数据结构。
- 能代表主流程的关键转折点。
- 能代表新增函数承担的核心职责。

默认应该尽量多 show code，尤其在下面几类位置：

- 结构变化的前后对照代码。
- 主流程入口和关键状态切换代码。
- 新增函数真正承接职责的函数体。
- 关键索引、绑定关系、持久化写入点。

不要只贴函数签名。能说明问题时，优先贴函数体里的关键分支、关键状态变更和关键副作用。

每段代码后都要说明：

- 这段代码在整个功能里的位置。
- 它解决了什么问题。
- 为什么这里需要新增这个函数或结构。

### 新增函数必须解释“为什么增加”

必须完整列出这个分支新增的所有函数，不能只挑几个顺眼的来讲。

最少要做这几步：

- 先给出“新增函数总清单”，不要遗漏任何新增函数。
- 再把这些函数分成 `核心函数`、`辅助函数`、`薄 helper` 三类。
- 每个函数都要说明它位于哪个文件、处在主链路的哪一段、为什么要增加。

不能只列函数名和职责。必须回答“为什么这里要新增这个函数，而不是继续写在原地”。

至少从下面几个角度解释：

- 它是不是新的真实入口。
- 它是不是统一了原先分散的生命周期或副作用。
- 它是不是承接了新数据结构的所有权。
- 它是不是为了让主流程能表达新的状态关系。

如果某个新增函数本质只是薄 helper，也要直说它的作用只是“降低局部重复”或“整理局部步骤”，不要把它讲成核心设计。

核心函数和辅助函数的区分也要讲明白：

- `核心函数`：真正承接新业务语义、关键状态变更、核心副作用或统一生命周期入口的函数。
- `辅助函数`：服务核心函数，但不单独决定主语义的函数，例如局部步骤拆分、参数组装、局部复用。
- `薄 helper`：主要做转发、包装、拆行或局部整理的函数。

如果新增函数很多，先给总表，再挑核心函数展开讲代码；但总表仍然必须完整。

### 给用户一个推荐阅读顺序

收尾时要告诉用户先看什么，再看什么。至少给出：

1. 最值得先看的 1 到 3 个文件。
2. 每个文件里最关键的 struct 或函数。
3. 推荐阅读顺序。

目标是让用户看完你的讲解，就能自己回到代码里继续读。

### 文档结构必须固定

这个 skill 生成和更新的文档，必须使用固定章节顺序。不要每次根据心情改结构，也不要因为本次内容少就省掉章节。

如果某一节当前没有太多内容，也保留章节标题，并在该节下写清楚当前信息较少或暂未涉及，不要直接删掉整节。

后续更新已有文档时：

- 继续沿用同一套章节顺序。
- 新内容优先补到对应章节下面。
- 不要随意把“结构变化”挪到“主流程”后面，也不要把“新增函数说明”拆散到别的地方。
- 只有在用户明确要求改模板时，才允许调整整体章节结构。

固定文档结构如下：

```md
## 文档导读

- 这篇文档适合谁读
- 建议按什么顺序读
- 如果只想快速了解，先看哪几节

## 功能背景

- 这个功能原本处在什么背景下
- 为什么需要理解这次改动

## 一句话结论

- 用一句话讲清楚这个分支做了什么
- 用一句话讲清楚实现上的核心变化

## 这个分支要解决什么问题

- 这次改动具体要解决什么痛点
- 用户或系统层面会遇到什么问题

## 原模型的问题在哪里

- 原有模型为什么难以支撑这个功能
- 旧结构、旧索引、旧流程分别卡在哪里

## 这次改动的核心原则

- 为什么要按现在这种方向改
- 这次改动坚持了哪些核心取舍

## 改动范围和文件地图

- 按职责分组列出主要改动文件
- 每个文件用一句话说明它在实现里的角色

## 核心概念速查

- 列出本分支理解成本最高的概念
- 每个概念用人话解释是什么、为什么需要、和谁有关

## 修改前结构和代码

- 贴修改前的核心 struct、字段、索引或入口代码
- 用人话解释旧结构怎么工作

## 修改后结构和代码

- 贴修改后的核心 struct、字段、索引或入口代码
- 用人话解释新结构怎么工作

## 结构变化对照

- 哪些代码删除了
- 哪些代码新增了
- 哪些字段、索引、职责发生了迁移
- 这里优先做前后对照说明

## 主流程图

- 第一张图先讲主路径
- 如果流程复杂，再补第二张展开图
- 使用 Mermaid

## 主流程代码讲解

- 从真实入口开始
- 路径里穿插关键代码片段，而不是只写步骤说明

## 新增函数总清单

- 完整列出这个分支新增的所有函数
- 标明函数所在文件

## 核心函数

- 逐个讲核心函数
- 每个函数附核心代码
- 每个函数都解释为什么要增加

## 辅助函数

- 逐个讲辅助函数
- 每个函数附核心代码
- 每个函数都解释为什么要增加

## 薄 helper

- 列出薄 helper
- 说明它们只是包装、转发还是局部整理
- 每个函数都解释为什么要增加

## 关键代码片段

- 优先充分贴代码，通常不少于 4 段
- 如果结构变化和流程变化都比较重，可以贴到 6 到 8 段
- 每段解释它为什么关键

## 推荐阅读顺序

- 先看哪些文件和哪些函数
```

如果用户要求讲得更细，就继续在这套固定结构下补代码片段，优先补结构前后代码、关键函数体和主流程转折点代码。不要只增加讨论，不 show code。

## 验证

最小检查通常包括：

```bash
git merge-base <base> <branch>
git diff --stat <base>...<branch>
git diff --name-only <base>...<branch>
```

需要深入某些结构和流程时再补：

```bash
git diff <base>...<branch> -- <key-files>
git log --oneline --decorate --no-merges <base>..<branch>
```

这个 skill 的验证目标是“讲解是否足以帮助用户理解实现”，不是“代码是否没有问题”。

完成后还要检查：

- 文档确实写入了 `/Users/chenmoyin/Documents/impl-doc/`。
- 如果是更新任务，更新的是原文档而不是平行副本。
- 最终回复里明确给出文档路径。
