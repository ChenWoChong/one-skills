# Branch Implementation Explainer

## 目标

这个 skill 不做 code review，不负责找 bug、回归风险、缺失测试或设计问题。它只做一件事：把一个分支的功能实现讲清楚。

输出必须让用户在几分钟内回答下面这些问题：

- 这个分支具体要实现什么功能。
- 为了实现它，哪些核心数据结构被修改了。
- 请求、事件或任务从哪里进，经过哪些关键函数，最后落到哪里。
- 哪些函数是新加的，为什么必须加，分别承担什么职责。
- 哪些代码是真正核心代码，值得先看；哪些只是辅助代码。

讲完以后，用户应该能直接定位核心文件、顺着主链路读代码，并把这个实现讲给别人听。

## 什么时候用

以下场景应触发这个 skill：

- “帮我讲清楚这个分支具体怎么实现的。”
- “我运行前想先快速理解这个功能的核心代码。”
- “告诉我这个分支改了哪些结构、流程和函数，方便我继续改。”
- “不要给我 code review，只讲实现细节。”
- “给我一个能快速看懂这段功能实现的讲解。”

如果用户明确要 review、挑 bug 或判断测试是否充分，不要只用这个 skill，应改用专门的 review 技能。

## 工作流程

1. 先确认比较范围。
2. 看 diff 和关键文件，判断这个分支的功能目标。
3. 先找数据结构变化，再找主流程，再找新增函数。
4. 从真实入口往下串主链路，不按文件顺序复述 diff。
5. 用结构图、流程图、关键代码片段把实现讲清楚。

## 规则

### 先确认比较范围

先搞清楚这个分支相对谁在变化。默认做法：

- 找 merge-base。
- 看 `git diff --stat` 和关键文件列表。
- 用提交信息、文档和测试名辅助判断功能目标。

如果比较基线错了，后面的结构讲解和流程讲解都会失真。

### 只讲实现，不做评判

这个 skill 的职责不是评审代码好坏，而是解释它怎么工作。

- 不输出 findings。
- 不主动评价“这里有 bug”“这里设计不好”。
- 不展开测试是否充分。
- 不把重点放在“还能怎么改”。

只有当某个实现选择本身会影响理解时，才说明它的取舍，例如“为什么把状态从 Session 挪到 LiveSession”。

### 先讲结构，再讲流程，再讲函数

不要按文件顺序复述 diff。固定顺序如下：

1. 这个分支实现了什么功能。
2. 修改了哪些核心数据结构或状态关系。
3. 主链路怎么走。
4. 新增或修改的关键函数分别做什么。
5. 哪些代码最值得先看。

先讲“骨架”，再讲“血流”，最后讲“器官”。不要一上来就贴零散函数。

### 数据结构变化必须讲修改前和修改后

只要分支为了实现功能修改了 struct、字段语义、索引关系、所有权、生命周期归属，就必须明确讲：

- 修改前结构是什么。
- 修改后结构是什么。
- 真正变化的是哪几个字段、索引或引用关系。
- 为什么这样改后，主流程更容易支撑这个功能。

如果只是说“新增了一个 struct”，但没有解释它替代了旧模型里的什么职责，这种讲解是不合格的。

### 必须给结构图

讲数据结构时，必须给一个简单结构图。优先用 Mermaid。

图里至少表现这些信息：

- 核心 struct 或对象有哪些。
- 它们之间谁持有谁、谁引用谁、谁负责索引谁。
- 哪些是持久化状态，哪些是运行时状态。

如果功能核心是“一个状态被拆成两个层次”或“多个索引围绕一个对象重建”，图里必须画出来。

### 必须给主流程图

讲主链路时，必须给一个流程图。优先用 Mermaid。

流程图不是装饰，至少要覆盖一条真实路径：

`入口 -> 参数组装 -> 核心对象定位/创建 -> 关键状态变化 -> 外部副作用 -> 返回/持久化`

如果这个功能至少有两条同等重要的路径，例如 `new/load` 和 `prompt`，可以画两条，但不要把所有边角情况都塞进一张图。

### 必须给核心代码片段

不能只讲抽象概念，必须贴少量关键代码片段，并解释为什么它们关键。

代码片段选择标准：

- 能代表新的核心数据结构。
- 能代表主流程的关键转折点。
- 能代表新增函数承担的核心职责。

不要贴大段无意义上下文。每段代码后都要说明：

- 这段代码在整个功能里的位置。
- 它解决了什么问题。
- 为什么这里需要新增这个函数或结构。

### 新增函数必须解释“为什么增加”

不能只列函数名和职责。必须回答“为什么这里要新增这个函数，而不是继续写在原地”。

至少从下面几个角度解释：

- 它是不是新的真实入口。
- 它是不是统一了原先分散的生命周期或副作用。
- 它是不是承接了新数据结构的所有权。
- 它是不是为了让主流程能表达新的状态关系。

如果某个新增函数本质只是薄 helper，也要直说它的作用只是“降低局部重复”或“整理局部步骤”，不要把它讲成核心设计。

### 给用户一个推荐阅读顺序

收尾时要告诉用户先看什么，再看什么。至少给出：

1. 最值得先看的 1 到 3 个文件。
2. 每个文件里最关键的 struct 或函数。
3. 推荐阅读顺序。

目标是让用户看完你的讲解，就能自己回到代码里继续读。

## 推荐输出结构

默认按这个顺序输出：

```md
## 功能概览

- 这个分支实现了什么。
- 这次实现的核心思想是什么。

## 结构变化

### 修改前
- 原来核心结构是什么

### 修改后
- 现在核心结构是什么

### 结构图
- 使用 Mermaid

## 主流程

### 核心链路
- 从真实入口讲 1 到 2 条路径

### 流程图
- 使用 Mermaid

## 关键代码

- 贴 2 到 4 段核心代码
- 每段解释它为什么关键

## 新增/修改函数说明

- 入口函数
- 关键函数
- 辅助函数
- 每个函数为什么增加

## 推荐阅读顺序

- 先看哪些文件和哪些函数
```

如果用户要求讲得更细，可以在“关键代码”下面继续展开，但不要把整段 diff 搬进来。

## 验证

最小检查通常包括：

```bash
git merge-base <base> <branch>
git diff --stat <base>...<branch>
git diff --name-only <base>...<branch>
```

需要深入某些结构和流程时再补：

```bash
git diff <base>...<branch> -- <key-files>
git log --oneline --decorate --no-merges <base>..<branch>
```

这个 skill 的验证目标是“讲解是否足以帮助用户理解实现”，不是“代码是否没有问题”。
