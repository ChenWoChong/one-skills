# Design Change Doc

## 目标

把一次代码模型、架构、链路或数据结构变更写成别人能快速判断的设计文档。文档必须让读者不读代码也能看懂：

- 现在是什么结构；
- 问题在哪里；
- 要改成什么结构；
- 具体改了哪些字段、索引、职责、链路和生命周期；
- 改完后如何满足需求；
- 有哪些非目标、兼容性和验证项。

## 工作流程

1. 先读相关代码和已有文档，确认真实结构，不凭记忆写方案。
2. 用一句话写清目标和验收条件，避免文档变成泛泛重构建议。
3. 写“当前问题”，直接指出旧结构导致的真实 bug、限制或理解成本。
4. 写“修改前结构”，包含简化图和核心代码结构。
5. 写“修改后结构”，包含简化图和核心代码结构。
6. 写“结构变化对比”，用表格逐项说明字段、索引、职责、链路、生命周期改了什么。
7. 写关键业务链路，例如 create/load/resume/prompt/close/update/reap 这类入口如何变化。
8. 写兼容性、非目标和验证项。
9. 最后自查：领导或评审人只看文档，是否能判断方案是否值得做、是否满足验收。

## 规则

### 必须写修改前结构

只写新方案是不够的。每份设计变更文档都要先把旧结构讲清楚：

- 旧 struct / map / DB 字段 / handler 链路是什么；
- 谁持有谁，谁索引谁，谁负责关闭或释放；
- 旧结构为什么无法满足需求；
- 如果有真实代码，贴最小代码片段，不要贴整文件；
- 如果是链路问题，用 `A -> B -> C` 或 Mermaid 图画出来。

### 必须写修改后结构

新结构要写成稳定模型，而不是只列改动点：

- 新增、删除、移动了哪些核心概念；
- 每个概念只负责什么，不负责什么；
- 新索引、新字段、新生命周期入口分别解决哪个问题；
- 新链路如何自然满足验收条件；
- 不要引入当前系统没有的概念，例如没有用户体系就不要加 `UserID`。

### 必须写对比说明

对比表是强制项，至少包含：

| 维度 | 修改前 | 修改后 | 为什么改 |
| --- | --- | --- | --- |
| 核心对象 | 旧对象职责 | 新对象职责 | 职责变化 |
| 索引关系 | 旧索引 | 新索引 | 查找链路变化 |
| 生命周期 | 旧关闭/清理语义 | 新关闭/清理语义 | 避免什么 bug |
| 业务链路 | 旧请求路径 | 新请求路径 | 如何满足验收 |

表格里的“为什么改”不能写空话，必须落到真实问题，例如串 session、恢复链路断、回收误伤、状态无法落库、索引粒度错误。

### 讲清职责和所有权

设计文档要明确“谁拥有数据，谁拥有副作用”：

- 持久化对象只放可恢复的逻辑状态；
- 当前进程运行态要单独说清楚，不要伪装成 DB 状态；
- socket / connection、session、runtime、store、handler、lifecycle 的边界要分开；
- 关闭连接不等于关闭业务会话，除非项目语义明确如此；
- 一个资源如果是 session 粒度，就不要挂到 connection 粒度。

### 链路说明要覆盖验收条件

如果用户给了验收条件，文档必须逐条回应。不要只说“支持”：

- 入口是什么；
- 查哪个索引；
- 状态在哪里创建或恢复；
- 出错时怎么处理；
- 后续请求为什么能继续走通；
- 哪个测试能证明这条验收条件成立。

### 非目标和兼容性不能省

设计文档必须明确不做什么：

- 不改协议；
- 不改 DB schema；
- 不引入新身份体系；
- 不保证某种恢复能力；
- 不扩大本次重构范围。

具体写什么取决于当前任务，不要机械照抄。非目标的价值是防止评审人误解方案边界。

### 文风

- 中文，直接，给工程负责人看得懂；
- 少用形容词，多写结构和链路；
- 不写“优化体验”“提升可维护性”这类没落点的话；
- 代码片段只保留关键字段和方法签名；
- 图用 Mermaid，复杂度控制在一屏内；
- 文档标题、章节名短而具体。

## 推荐结构

```markdown
# Xxx Design

## 目标

## 当前问题

## 修改前结构

## 设计

### 修改后结构

### 结构变化对比

### Manager / Store / Handler 等关键模块变化

## 关键链路

## 关闭和回收语义

## 改动范围

## 兼容性

## 非目标

## 验证

## 实施顺序
```

章节可以按任务裁剪，但“修改前结构 / 修改后结构 / 结构变化对比 / 验证”不能省。

## 验证

交付前检查：

- 文档是否说明了修改前结构；
- 文档是否说明了修改后结构；
- 是否有对比表讲清楚改了什么；
- 是否逐条回应用户给出的验收条件；
- 是否避免引入当前系统没有的概念；
- 是否写了兼容性、非目标和验证项；
- 如果修改了项目文档，运行对应仓库的文档或 skill 校验命令。
